# FinanceOS — Database Performance Considerations (Phase 0 + Phase 1 + Phase 2)

- **`Transaction` indexes**: `@@index([userId, date])` supports the dashboard's date-ranged aggregation queries (monthly summary, trends) and the transaction table's default sort (most recent first) in one index. `@@index([accountId])` and `@@index([categoryId])` support filtering by account/category in the transaction table without a full scan.
- **Pagination is offset-based (`page`/`pageSize`)** in Phase 1 per api-contracts.md — acceptable at expected per-user transaction volumes (thousands, not millions, of rows per user). Performance Engineer should revisit with a cursor-based approach only if profiling in a later phase shows offset pagination degrading (typically only relevant at very deep page numbers, unlikely for a personal-finance-scale dataset).
- **`Decimal(14,2)` for money fields**: avoids floating-point rounding errors on financial amounts; every sum/aggregation (net worth, monthly totals) must be done in Postgres or via a Decimal-safe library, never native JS floating point.
- **Dashboard aggregation queries** (`getNetWorth`, `getMonthlySummary`, `getSpendingByCategory`, `getMonthlyTrends`) should use Prisma's `aggregate`/`groupBy` to push summation to Postgres rather than fetching all transactions and summing in application code — this is a Backend Engineer implementation requirement, flagged here because it's a common mistake that only shows up as a problem once a user has months of transaction history.
- **CSV import** (potentially hundreds of rows in one request) should batch-insert via `createMany` after in-memory dedup, not one `create` call per row, to avoid N round-trips to the database.
- No caching layer (Redis, etc.) is introduced in Phase 0/1 — dashboard queries are cheap enough at this data scale to compute on each request. Revisit only if Performance Engineer's Phase 1 review shows otherwise.

## Phase 2 additions

- **`BillOccurrence` indexes**: `@@index([userId, dueDate])` supports both the upcoming-list query (userId + dueDate ascending, limit N) and the calendar-month view (userId + dueDate range) with one index — mirrors the same `[userId, date]` pattern already used on `Transaction`. `@@unique([billId, dueDate])` doubles as the idempotency guard for lazy occurrence generation (an `upsert`/`createMany ... skipDuplicates` on this key is the intended access pattern, not a separate existence check before insert).
- **Lazy occurrence generation must be bounded, not unbounded-lookback**: `ensureOccurrencesGenerated` should only generate occurrences up to a fixed forward horizon (e.g. next N months) on each read, not attempt to backfill or generate arbitrarily far into the future in one request. Generating a large horizon in a single request for a bill with a long-idle history is the Phase 2 equivalent of the CSV-import-size risk already flagged in Phase 1 — bound it explicitly rather than letting one slow request emerge from an edge case.
- **`Notification` polling** (`GET /api/notifications`, called on an interval per the notification-bell's ambient-polling design): `@@index([userId, readAt])` supports "unread count for this user" as an index-only scan; `@@index([userId, createdAt])` supports the inbox's default sort. `ensureNotifications`'s upsert-based materialization (relying on the `@@unique([budgetCategoryId, type])` / `@@unique([billOccurrenceId, type])` constraints) should run as part of the same request as the poll, not as a separate scheduled job — there is no background-job infrastructure in this app, and the read volume here (one user's own budget categories + upcoming bill occurrences, both already small, bounded sets) doesn't warrant one.
- **`Budget`/`BudgetCategory` are read per month, not aggregated across history** in the common case (Budget Health Score, the planner view) — `@@unique([userId, month])` and `@@index([budgetId, categoryId])`-equivalent (via the `@@unique([budgetId, categoryId])` constraint, which Postgres also uses as a lookup index) keep the current-month read a small, indexed fetch regardless of how many months of history a long-time user accumulates.
- **`Decimal(14,2)` continues to apply** to every new money column (`Budget`/`Goal`/`Bill` amounts) — same rounding-safety rationale as Phase 1.
