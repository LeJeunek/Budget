# FinanceOS — Database Performance Considerations (Phase 0 + Phase 1 + Phase 2 + Phase 3a + Phase 3b + Phase 4a)

- **`Transaction` indexes**: `@@index([userId, date])` supports the dashboard's date-ranged aggregation queries (monthly summary, trends) and the transaction table's default sort (most recent first) in one index. `@@index([accountId])` and `@@index([categoryId])` support filtering by account/category in the transaction table without a full scan.
- **Pagination is offset-based (`page`/`pageSize`)** in Phase 1 per api-contracts.md — acceptable at expected per-user transaction volumes (thousands, not millions, of rows per user). Performance Engineer should revisit with a cursor-based approach only if profiling in a later phase shows offset pagination degrading (typically only relevant at very deep page numbers, unlikely for a personal-finance-scale dataset).
- **`Decimal(14,2)` for money fields**: avoids floating-point rounding errors on financial amounts; every sum/aggregation (net worth, monthly totals) must be done in Postgres or via a Decimal-safe library, never native JS floating point.
- **Dashboard aggregation queries** (`getNetWorth`, `getMonthlySummary`, `getSpendingByCategory`, `getMonthlyTrends`) should use Prisma's `aggregate`/`groupBy` to push summation to Postgres rather than fetching all transactions and summing in application code — this is a Backend Engineer implementation requirement, flagged here because it's a common mistake that only shows up as a problem once a user has months of transaction history.
- **CSV import** (potentially hundreds of rows in one request) should batch-insert via `createMany` after in-memory dedup, not one `create` call per row, to avoid N round-trips to the database.
- No caching layer (Redis, etc.) is introduced in Phase 0/1 — dashboard queries are cheap enough at this data scale to compute on each request. Revisit only if Performance Engineer's Phase 1 review shows otherwise.

## Phase 2 additions

- **`BillOccurrence` indexes**: `@@index([userId, dueDate])` supports both the upcoming-list query (userId + dueDate ascending, limit N) and the calendar-month view (userId + dueDate range) with one index — mirrors the same `[userId, date]` pattern already used on `Transaction`. `@@unique([billId, dueDate])` doubles as the idempotency guard for lazy occurrence generation (an `upsert`/`createMany ... skipDuplicates` on this key is the intended access pattern, not a separate existence check before insert).
- **Lazy occurrence generation must be bounded, not unbounded-lookback**: `ensureOccurrencesGenerated` should only generate occurrences up to a fixed forward horizon (e.g. next N months) on each read, not attempt to backfill or generate arbitrarily far into the future in one request. Generating a large horizon in a single request for a bill with a long-idle history is the Phase 2 equivalent of the CSV-import-size risk already flagged in Phase 1 — bound it explicitly rather than letting one slow request emerge from an edge case.
- **`Notification` polling** (`GET /api/notifications`, called on an interval per the notification-bell's ambient-polling design): `@@index([userId, readAt])` supports "unread count for this user" as an index-only scan; `@@index([userId, createdAt])` supports the inbox's default sort. `ensureNotifications`'s upsert-based materialization (relying on the `@@unique([budgetCategoryId, type])` / `@@unique([billOccurrenceId, type])` constraints) should run as part of the same request as the poll, not as a separate scheduled job — there is no background-job infrastructure in this app, and the read volume here (one user's own budget categories + upcoming bill occurrences, both already small, bounded sets) doesn't warrant one.
- **`Budget`/`BudgetCategory` are read per month, not aggregated across history** in the common case (Budget Health Score, the planner view) — `@@unique([userId, month])` and `@@index([budgetId, categoryId])`-equivalent (via the `@@unique([budgetId, categoryId])` constraint, which Postgres also uses as a lookup index) keep the current-month read a small, indexed fetch regardless of how many months of history a long-time user accumulates.
- **`Decimal(14,2)` continues to apply** to every new money column (`Budget`/`Goal`/`Bill` amounts) — same rounding-safety rationale as Phase 1.

## Phase 3a additions

- **`Debt` has no dedicated composite index beyond `@@index([userId])` and `accountId`'s own unique index.** Per `debt-tracker.md`'s own framing ("a dozen credit cards and loans" as the "very high" case), this is a small, per-user, non-paginated list — the same volume class as `Goal`, not `Transaction`. The Net Worth aggregation's `unlinkedDebtLiability` query (`WHERE userId = ? AND archivedAt IS NULL AND accountId IS NULL AND balance > 0`) is a single-table indexed filter on a small row count per user; no additional index is warranted unless a future Performance Engineer review finds otherwise at a data scale this schema doesn't currently anticipate.
- **`Holding` allocation/portfolio-overview queries (`getAllocation`, `getPortfolioOverview`) scope by `accountId` (via `@@index([accountId])`) or by `userId` directly (`@@index([userId])`, for the portfolio-wide, cross-container view)** — both are cheap, indexed lookups. **`HoldingValueHistoryEntry`'s `@@index([holdingId, recordedAt])`** is the one Phase 3a index specifically sized for potential growth: `getGrowthHistory` reads this table ordered by `recordedAt` per holding, and per investments.md's own Definition of Done ("remains responsive... with... a long value-update history"), a holding a user updates weekly for years accumulates hundreds of rows — still trivial for a single indexed range scan, but flagged here as the one Phase 3a table whose row count per entity is expected to actually grow over time (unlike `Debt`/`IncomeStream`, which stay small and mostly static per user).
- **The Investments derived-balance write-back (`Account.balance` recalculated on every `Holding` create/update/close) must run inside the same Prisma `$transaction` as the holding mutation itself** (Architecture.md's binding requirement, restated here for the Performance Engineer's review) — this is a correctness requirement, not a performance one, but it does mean every holding-current-value edit costs one extra indexed `UPDATE` on `Account` in the same round trip. At this feature's expected holding-count-per-container (tens, not thousands), this is negligible; it would only become a concern if a future phase introduced bulk/CSV holding import, which is explicitly out of Phase 3a's scope.
- **`IncomeOccurrence` mirrors `BillOccurrence`'s indexing exactly**: `@@index([userId, expectedDate])` supports both the expected-upcoming-income query (userId + expectedDate ascending, limit N) and per-stream history range queries; `@@unique([streamId, expectedDate])` doubles as the lazy-generation idempotency guard, same `upsert`/`createMany ... skipDuplicates` access pattern already established for `BillOccurrence`. `IrregularIncomeEvent`'s `@@index([userId, date])` supports its own chronological history view without needing the occurrence-generation idempotency guard (it has no generated rows to dedupe against).
- **The cross-table Bills<->Recurring Income exclusivity check (`lib/transaction-link-guard.ts`) reads two additional indexed columns before every link action**: `BillOccurrence.transactionId` and `IncomeOccurrence`/`IrregularIncomeEvent.transactionId`, all three already `@unique` (and therefore already indexed) for their own single-table constraint. The guard's query is `SELECT 1 FROM bill_occurrence WHERE transaction_id = ? UNION ALL SELECT 1 FROM income_occurrence WHERE transaction_id = ? UNION ALL SELECT 1 FROM irregular_income_event WHERE transaction_id = ?` (or the Prisma equivalent, three single-row indexed point lookups) — cheap regardless of table size, since it's an equality lookup on a unique-indexed column, not a scan. No new index is needed specifically for the guard.
- **`NetWorthSnapshot`'s `@@unique([userId, capturedDate])` and `@@index([userId, capturedAt])`**: the unique constraint is both the cron's idempotency guard (upsert-per-day, not a second existence check before insert — same pattern already established for `BillOccurrence`/`IncomeOccurrence`'s lazy-generation idempotency) and the natural point-lookup for "has today's snapshot already been captured for this user." The separate `capturedAt` index exists for Phase 3b's future ordered range queries ("net worth over the last N months") without forcing that later read to filter on the truncated date column. **Flagged for the Performance Engineer's Phase 3a review** (already flagged once in api-contracts.md, restated here from the schema's own angle): `captureAllUsersNetWorthSnapshots()` runs one `getNetWorth` computation per user in the entire system on every cron invocation — at this app's current scale this is a small, bounded loop, but if the user base grows large enough for a single invocation to run long, the fix is cursor-based batching within that function (process N users per invocation), not a schema change; `NetWorthSnapshot` itself needs no redesign to support that later optimization.
- **No caching layer is introduced for any Phase 3a read path** (allocation, growth history, payoff projections, expected-upcoming-income) — consistent with the Phase 0/1 decision above; all of these are single-user-scoped, indexed, small-row-count queries at this app's expected data scale. Revisit only if a Performance Engineer review of a later phase shows otherwise.

## Phase 3b additions

- **`DismissedSubscriptionMerchant`'s `@@unique([userId, normalizedMerchantName])` is both the write-path idempotency guard (a re-dismissal is an `upsert`, not a second row or an existence check first) and the read-path lookup Subscription Cost Detection needs on every run** (`WHERE userId = ?` — a full per-user scan of this table, which stays small: at most one row per merchant a user has ever dismissed, realistically dozens at most). No composite/covering index is needed beyond `@@index([userId])`; this table's row count per user is bounded by the same small-list class as `Debt`/`IncomeStream`, not `Transaction`.
- **`FinancialGoal.@@index([linkedDebtId])` supports the Debt Payoff exclusivity guard's own pre-create lookup** (`WHERE linkedDebtId = ? AND archivedAt IS NULL`, run inside the same Prisma `$transaction` as the create — see `docs/database/er-diagram.md`'s Phase 3b design notes for the full application-level-guard-vs-partial-index reasoning) — a cheap, indexed, single-row-expected lookup, not a scan. **Flagged as a monitored risk, not a database-enforced guarantee**, the same status already accepted for Phase 3a's Bills↔Recurring-Income cross-table guard: the realistic race window (a single authenticated user double-submitting "create" within milliseconds of themselves) is narrow enough that this Architect judged an application-level check sufficient rather than introducing this schema's first hand-edited partial-unique-index migration. Revisit only if this is ever actually hit in production (per the same evidence-driven standard the Phase 3a precedent set), not preemptively.
- **`FinancialGoal.@@index([userId])` alone is sufficient for `getFinancialGoals`'s list query** — per financial-goals.md's own framing (a handful of milestones per user, the same small-list volume class as `Debt`/`Goal`/`IncomeStream`), no composite index (e.g. `[userId, archivedAt]`) is warranted at this feature's expected scale; the active/archived split is a small in-memory filter over an already-tiny per-user row set, not a query-plan concern.
- **`FinancialGoalAccount`'s composite primary key (`@@id([financialGoalId, accountId])`) already serves as its own lookup index** for both directions this table is ever queried from: "which Accounts does this goal's subset include" (`WHERE financialGoalId = ?`, a leading-column match against the composite PK) and the reverse existence check an Account-archival flow might eventually want. No additional index is needed — this table's total row count across a user's entire account history is bounded by (goals × accounts-per-subset), trivially small.
- **Financial Goals' read path is entirely downstream of already-optimized queries in other modules** (`debt.service.getDebtById` — a single indexed by-id lookup; `dashboard.service.getNetWorth`/`getMonthlySummary` — already-reviewed Phase 1/3a aggregations; `accounts.service.getAccounts` — an indexed `userId` scan), so `getFinancialGoals`'s own cost is dominated by however many of these downstream calls a user's mix of goal types triggers (at most one call per goal, per type, per read) — bounded by the same "handful of goals per user" scale as the table itself, not a new performance surface of its own.
- **Analytics' 11 metrics introduce no new indexes** — Risk #11 (resolved in `docs/architecture/Architecture.md`'s Phase 3b section: raw on-read aggregation, no materialized/cached aggregates) relies entirely on `Transaction`'s existing `@@index([userId, date])`/`@@index([categoryId])`, `Budget`/`BudgetCategory`'s existing indexes, and `HoldingValueHistoryEntry`'s existing `@@index([holdingId, recordedAt])` (for Savings Growth's `getGainLossForPeriod`, which sums this table's rows within a period — a range scan this index already supports without modification). **Category Trends and Subscription Cost Detection are flagged, per Architecture.md's own Risk #11 writeup, as the two metrics most likely to need a future query-shape refinement** (a raw `date_trunc` SQL grouped query for Category Trends' two-dimensional category×month bucketing; possibly a covering index on `Transaction(userId, merchant, date)` for Subscription Detection's full-history merchant scan, if profiling at real data volumes ever shows the existing `[userId, date]` index insufficient for that specific access pattern) — **not added speculatively here**, consistent with this document's evidence-first stance everywhere else; revisit only if a Performance Engineer review of real production volumes finds a specific metric measurably slow.
- **No caching layer is introduced for any Phase 3b read path** (Net Worth History, any of the 11 Analytics metrics, Financial Goals) — consistent with the Phase 0/1/3a decision above, and Architecture.md's own explicit Risk #11 re-justification (not just an inherited default) reaching the same conclusion for this specific phase's read patterns.

## Phase 4a additions

- **`CategorySuggestion`'s at-most-one-`PENDING`-suggestion-per-transaction rule is a database-enforced constraint, not just a monitored application-level guard** — a deliberate departure from Phase 3a's Bills↔Recurring-Income cross-table exclusivity and Phase 3b's Debt Payoff exclusivity (`er-diagram.md`'s Phase 3b design notes), both of which remain application-level-only. This table's AUTOMATIC path is written by a cron route capable of overlapping invocations racing the same check-then-create guard from separate execution contexts — a risk profile the two prior precedents' "one user, within milliseconds of themselves" framing never covered — so a Security Architect Phase 4a design-stage review finding (Finding 5) prompted adding a hand-authored partial unique index, `CREATE UNIQUE INDEX ... ON category_suggestion ("transactionId") WHERE status = 'PENDING'` (migration `20260722193327_category_suggestion_pending_partial_unique_index`), rather than leaving this as a third monitored-but-unenforced instance of the pattern. See `er-diagram.md`'s Phase 4a design notes for the full reasoning, including why a run-level cron idempotency key (mirroring `NetWorthSnapshot`'s `(userId, capturedDate)` pattern) was considered and rejected in favor of the partial index. `@@index([transactionId, status])` remains the index the application-level fast-path guard's own pre-create lookup (`WHERE transactionId = ? AND status = 'PENDING'`) relies on for a cheap, single-row-expected indexed point-lookup — that guard is kept as a fast-path that avoids a wasted model call in the common (non-racing) case, but the partial unique index, not this composite index, is now the actual enforcement mechanism; a second writer that loses the race gets a Prisma `P2002` error the application code must catch and treat as an idempotent no-op. `@@index([userId, source, status])` supports the Success Metrics' own required query shapes (the automatic-path-only acceptance-rate computation, and the rejection-rate computation, both need to filter/count by `(userId, source, status)` together) without a full-table scan. Row volume per user is bounded by transaction volume (this app's largest table), but each individual query against this table is always scoped by an indexed prefix (`transactionId` or `userId`), never an unscoped scan.
- **`BudgetAdvisorCache` and `SpendingInsightsCache` are both bounded, single-row-per-key tables by design** — `@@unique([userId, month])` / `@@unique([userId, period])` guarantee at most one row per user per key, so neither table's total size ever grows unbounded per user the way an append-only ledger would; both are upserted in place on every generation attempt (successful or not), never appended to. `@@index([userId, month])` / `@@index([userId, period])` (mirroring `Budget`'s own `@@unique` + `@@index` pair on the identical columns) serve every lookup this feature needs — the implicit generate-on-first-view read, the explicit Refresh action's rate-limit check, and the upsert itself — as a single indexed point-lookup, never a scan.
- **`MonthlySummary` is a genuine, permanently-growing history table** (one row per user per fully-closed calendar month, forever) — but its growth rate is the lowest-bandwidth of any table in this schema (at most 12 rows per user per year, the same "small, bounded, non-`Transaction`-scale" volume class as `Debt`/`Goal`/`IncomeStream`, not a concern even over a many-year-old account). `@@unique([userId, month])` + `@@index([userId, month])` (identical shape to `Budget`'s own pair) serve both `getMostRecentSummary` (`ORDER BY month DESC LIMIT 1`) and `getSummaryHistory` (`ORDER BY month`) as a single indexed scan bounded by this table's own naturally tiny per-user row count.
- **`FinancialHealthScoreSnapshot` mirrors `NetWorthSnapshot`'s indexing and cost profile exactly** — `@@unique([userId, capturedDate])` is both the cron's idempotency guard (upsert-per-day, identical pattern to `NetWorthSnapshot`/`BillOccurrence`/`IncomeOccurrence`'s own lazy-generation idempotency) and the natural "did we already snapshot this user today" point-lookup; `@@index([userId, capturedAt])` serves the historical-trend sparkline's ordered range query. **The same cron-fan-out risk already flagged for `NetWorthSnapshot`'s cron applies here identically, restated rather than newly introduced**: `app/api/cron/financial-health-score-snapshot/route.ts` runs one score computation (four cross-domain reads: Debt, Recurring Income, Budgeting's `getBudgetHealthScore`, Dashboard's `getNetWorth`) plus one narrative-generation call per user in the entire system, per invocation — at this app's current scale a small, bounded loop, but the mitigation, if ever needed, is the same one already on record for `NetWorthSnapshot`: cursor-based batching within the capture function (process N users per invocation), not a schema change. No `FinancialHealthScoreSnapshot` redesign is needed to support that later optimization, exactly as already noted for `NetWorthSnapshot`.
- **No new caching-layer precedent is introduced by `BudgetAdvisorCache`/`SpendingInsightsCache`**, restated from the schema's own angle (full reasoning in `er-diagram.md`'s Phase 4a design notes): Risk #11 declined caching *deterministic* Prisma aggregation, which stays uncached everywhere in this app, including the Financial Health Score's own `service.ts` formula (a live, on-read computation with no cache of its own, exactly like `getBudgetHealthScore`/`getNetWorth`). These two tables cache *non-deterministic, non-free-to-regenerate AI output* instead — a different problem Risk #11 never evaluated and didn't decline.
- **No new index is added to `Transaction`, `Category`, or any other pre-existing table for Phase 4a** — every one of `CategorySuggestion`'s foreign keys (`transactionId`, `suggestedCategoryId`) reads against columns (`Transaction.id`, `Category.id`) already indexed by their own primary keys; the categorization cron's own read query (uncategorized transactions with no existing `PENDING` suggestion) is bounded by `Transaction`'s existing `@@index([userId, date])`/a `categoryId IS NULL` filter and `CategorySuggestion`'s own `@@index([transactionId, status])`, needing nothing new on `Transaction`'s side.
