# FinanceOS — Risk Register

| # | Risk | Phase | Likelihood | Impact | Mitigation | Owner |
|---|------|-------|-----------|--------|------------|-------|
| 1 | Transaction/Account schema designed too narrowly in Phase 1, forcing breaking migrations when Debt/Investments (Phase 3a) need to reuse "account-like" concepts | 1→3a | High | High | Database Architect modeled `Account` with a `type` discriminator and shared balance/interest fields from the start. Resolved at 3a kickoff: Database Architect decided and documented the Debt/Investment linkage approach in `docs/database/` before 3a backend work began. **Resolved** — 3a shipped and passed its review gate. | Database Architect |
| 2 | AI transaction categorization / insights are vulnerable to prompt injection via user-controlled text (merchant names, transaction notes) | 4a | Medium | Medium | Treat all transaction text as untrusted input to the model; structured-output schema validation (Zod) on every AI response before it touches the database; Security Architect review required before release. **Extended for 4a kickoff (2026-07-21):** this is the first AI/LLM integration this codebase has ever had, so the mitigation now explicitly includes a *design-stage* Security Architect review (before backend implementation starts), in addition to the standing pre-release review — see `roadmap.md`'s Phase 4a milestones. | AI Engineer, Security Architect |
| 3 | CSV import (Phase 1) is a common source of malformed data, encoding issues, and duplicate-transaction bugs | 1 | High | Medium | Backend Engineer validates and dedupes on import; Bug Hunter specifically stress-tests large/malformed CSVs before Phase 1 sign-off. **Resolved** — Phase 1 shipped and passed its review gate. | Backend Engineer, Bug Hunter |
| 4 | Financial data (balances, transactions) is high-sensitivity; any authorization bug exposes one user's finances to another | All | Low | Critical | Every API route/Server Action scoped by authenticated user ID at the query level (never trust client-supplied user/account IDs alone); Security Architect reviews every phase before release, not just at the end. **Extended for Phase 4b (2026-07-21):** this same discipline now explicitly covers PDF report generation/download (a user must never be able to generate or fetch another user's report) and outbound notification email content — see Risk #17. **Extended for Phase 4c:** now also covers admin-side data views (view users, audit logs) — see Risk #18. | Backend Engineer, Security Architect |
| 5 | PDF report generation (Phase 4) and large transaction tables (Phase 1+) risk becoming a performance/memory bottleneck as data grows | 1, 4b | Medium | Medium | Pagination and server-side filtering from day one in Phase 1 (not retrofitted); Performance Engineer reviews before each phase's release. This concern is concretely realized by Phase 3b's Analytics suite (see Risk #11) and again by Phase 4b's PDF generation. **Extended for 4b kickoff (2026-07-21):** the deployment target is serverless (Vercel, per Phase 0), which constrains PDF-library choice — headless-browser-based rendering carries cold-start, function-size, and execution-time limits in that environment that a non-browser JS-native renderer would not. Flagged as a binding constraint for the Solution Architect's 4b library choice, not just a post-hoc performance review item — see `roadmap.md`'s Phase 4b section. | Performance Engineer, Solution Architect |
| 6 | Feature scope creep — roadmap is already large; pressure to pull later-phase features into an earlier phase for a "richer demo" | All | High | High | Roadmap phase boundaries are binding; CTO approval required to move any feature to an earlier phase or across the 3a/3b boundary (see Risk #13) or the 4a/4b/4c boundaries (see Risk #16) | CTO |
| 7 | Better Auth is a comparatively newer library vs NextAuth; ecosystem/adapter gaps could block Phase 0 | 0 | Low | Medium | Time-boxed Phase 0 spike; fall back to NextAuth if Better Auth blocks progress for more than 1 day. **Resolved** — Better Auth shipped cleanly in Phase 0, no fallback needed. | Backend Engineer |
| 8 | Timezone/date handling bugs in bills, budgets (monthly resets), and recurring schedules | 2 | Medium | Medium | Store all dates in UTC, compute "due today"/"month boundary" logic against the user's stored timezone preference, not server time; explicit Bug Hunter pass on this before Phase 2 sign-off. **Resolved** — Phase 2 shipped and passed its review gate. Note for Phase 4c: the Customization spec's "themes/currency display" work should not silently introduce a second timezone-preference source — Solution Architect should confirm the existing preference storage (from Phase 2) is reused, not duplicated. | Backend Engineer, Bug Hunter |
| 9 | Debt/Investments schema shape (standalone models vs. extending the existing `Account` type discriminator, vs. a hybrid) decided hastily could force a breaking migration later once 3b's Net Worth aggregation and Analytics depend on it | 3a | Medium | High | Explicit Database Architect design decision as its own reviewed milestone before any Phase 3a backend work starts (not folded silently into implementation) — document the chosen approach and rationale in `docs/database/` before backend implementation begins. **Resolved** — decided and documented before 3a backend work began; 3a and 3b both shipped successfully on top of it. | Database Architect |
| 10 | Net Worth History chart (3b) has no historical data to chart if periodic snapshotting only begins when the chart UI itself is built | 3a→3b | High | Medium | Begin a lightweight periodic Net Worth Snapshot capture (backend only, no UI) at the end of Phase 3a — see `roadmap.md`'s Phase 3a milestones — so 3b's chart launches with real historical data already accumulated instead of an empty state. **Resolved** — shipped as scoped; the same "don't build the empty-history feature first" reasoning is reused for Phase 4c's Admin audit log (see Risk #16's split rationale in `roadmap.md`). | Backend Engineer |
| 11 | Analytics suite (3b) is compute-heavy over full Transaction/Account/Debt/Investment history (yearly spend, heatmaps, growth trends) — the general concern already flagged in Risk #5, now realized in concrete form | 3b | Medium | Medium | Performance Engineer reviews query/aggregation design before backend implementation begins (not only at phase-end release review); Solution Architect evaluates materialized/cached aggregate tables during the 3b architecture pass if live-query performance is a concern at realistic data volumes. **Resolved** — 3b shipped and passed its review gate. | Performance Engineer, Solution Architect |
| 12 | Financial Goals (broad, 3b) and Savings Goals (Phase 2) are conceptually adjacent — risk of user-facing duplication/confusion or two competing "goal" UIs if the boundary isn't explicit | 3b | Medium | Medium | Product Owner must define the explicit boundary and relationship between the two models in the 3b spec, before Database Architect schema work begins for `FinancialGoal`. **Resolved** — see `docs/product/financial-goals.md`'s explicit boundary decision; 3b shipped on top of it. | Product Owner |
| 13 | Splitting Phase 3 into 3a/3b creates a temptation to blur the boundary (e.g., pulling an Analytics metric into 3a "since it's related," or a Debt/Investment field into 3b) | 3a, 3b | Medium | Medium | The 3a/3b boundary defined in `roadmap.md` is binding, same as Risk #6; CTO approval required to move any feature across it. **Resolved** — no cross-boundary moves occurred; v1 shipped with the boundary intact. | CTO |
| 14 | Financial Health Score (Phase 4a) scope is ambiguous — an LLM-computed judgment vs. a deterministic formula over existing Analytics/Debt/Budget data with an optional AI-generated narrative — and guessing wrong misassigns module ownership (AI Engineer vs. Backend Engineer) before schema/architecture work | 4a | Medium | Medium | Product Owner must resolve this explicitly in the 4a spec, before the Solution Architect + AI Engineer architecture pass — see `roadmap.md`'s Phase 4a open scope question, following the same discipline Risk #12 established for the SavingsGoal/FinancialGoal boundary | Product Owner |
| 15 | LLM/AI provider selection (Phase 4a) carries data-privacy (transaction/merchant text sent to a third party), cost, latency, and vendor-lock-in dimensions — this codebase has no existing AI/LLM integration to build on, so there is no established pattern to reuse or fall back to | 4a | Medium | High | Solution Architect + AI Engineer make this decision jointly during 4a's architecture pass, bound by CTO-set constraints: structured/validated output only, minimize data sent to the model, isolate the integration behind a swappable module boundary in `lib/ai/`, and keep cost/latency bounded and observable — see `roadmap.md`'s Phase 4a section. Not a decision to leave implicit inside implementation. | Solution Architect, AI Engineer |
| 16 | Splitting Phase 4 into 4a/4b/4c creates the same boundary-blur temptation Risk #13 flagged for 3a/3b (e.g., pulling Notifications v2's email delivery into 4a "since it's related to AI summaries," or an Admin screen into 4b) | 4a, 4b, 4c | Medium | Medium | The 4a/4b/4c boundaries defined in `roadmap.md` are binding, same as Risk #6/#13; CTO approval required to move any feature across a boundary | CTO |
| 17 | Notifications v2's email delivery (4b) is this codebase's first outbound email infrastructure and its first feature that sends financial content (balances, summaries) outside the app boundary entirely — a new data-egress and compliance surface (deliverability, unsubscribe/preference handling, PII in email body/subject) that the standing "no cross-user leakage" discipline (Risk #4) wasn't originally written with in mind | 4b | Medium | High | Solution Architect + Backend Engineer select and integrate the email provider within CTO-set constraints (see `roadmap.md`'s Phase 4b section); Security Architect review before release focused specifically on email content/PII exposure and unsubscribe/preference compliance, distinct from and in addition to the standard authorization review every feature gets | Backend Engineer, Security Architect |
| 18 | Admin (4c) introduces this codebase's first privilege/authorization tier — no `role` or permission concept exists on `User` today — creating a new class of risk (privilege escalation / unauthorized access to other users' data via admin views) beyond the "wrong user, right role" model every prior phase's authorization checks were built around | 4c | Medium | High | Solution Architect + Database Architect decide the admin-authorization mechanism as its own explicit, reviewed milestone before any Admin backend work starts (same discipline Risk #9 required for 3a's Account-linkage decision — see `roadmap.md`'s Phase 4c section); Security Architect review is the headline focus of 4c's release gate, covering privilege escalation, audit-log tamper-resistance, and feature-flag access control specifically | Database Architect, Solution Architect, Security Architect |
