# FinanceOS — Risk Register

| # | Risk | Phase | Likelihood | Impact | Mitigation | Owner |
|---|------|-------|-----------|--------|------------|-------|
| 1 | Transaction/Account schema designed too narrowly in Phase 1, forcing breaking migrations when Debt/Investments (Phase 3) need to reuse "account-like" concepts | 1→3 | High | High | Database Architect models `Account` with a `type` discriminator and shared balance/interest fields from the start, even though credit/loan/investment-specific fields aren't used until Phase 3 | Database Architect |
| 2 | AI transaction categorization / insights are vulnerable to prompt injection via user-controlled text (merchant names, transaction notes) | 4 | Medium | Medium | Treat all transaction text as untrusted input to the model; structured-output schema validation (Zod) on every AI response before it touches the database; Security Architect review required before release | AI Engineer, Security Architect |
| 3 | CSV import (Phase 1) is a common source of malformed data, encoding issues, and duplicate-transaction bugs | 1 | High | Medium | Backend Engineer validates and dedupes on import; Bug Hunter specifically stress-tests large/malformed CSVs before Phase 1 sign-off | Backend Engineer, Bug Hunter |
| 4 | Financial data (balances, transactions) is high-sensitivity; any authorization bug exposes one user's finances to another | All | Low | Critical | Every API route/Server Action scoped by authenticated user ID at the query level (never trust client-supplied user/account IDs alone); Security Architect reviews every phase before release, not just at the end | Backend Engineer, Security Architect |
| 5 | PDF report generation (Phase 4) and large transaction tables (Phase 1+) risk becoming a performance/memory bottleneck as data grows | 1, 4 | Medium | Medium | Pagination and server-side filtering from day one in Phase 1 (not retrofitted); Performance Engineer reviews before each phase's release | Performance Engineer |
| 6 | Feature scope creep — 4-phase roadmap is already large; pressure to pull Phase 3/4 features into Phase 1 for a "richer demo" | All | High | High | Roadmap phase boundaries are binding; CTO approval required to move any feature to an earlier phase | CTO |
| 7 | Better Auth is a comparatively newer library vs NextAuth; ecosystem/adapter gaps could block Phase 0 | 0 | Low | Medium | Time-boxed Phase 0 spike; fall back to NextAuth if Better Auth blocks progress for more than 1 day | Backend Engineer |
| 8 | Timezone/date handling bugs in bills, budgets (monthly resets), and recurring schedules | 2 | Medium | Medium | Store all dates in UTC, compute "due today"/"month boundary" logic against the user's stored timezone preference, not server time; explicit Bug Hunter pass on this before Phase 2 sign-off | Backend Engineer, Bug Hunter |

Update this register at the start of each phase's architecture review — add newly discovered risks, close ones that are no longer applicable, and do not delete rows (mark them resolved instead) so the history stays visible to the whole team.
