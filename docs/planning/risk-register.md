# FinanceOS — Risk Register

| # | Risk | Phase | Likelihood | Impact | Mitigation | Owner |
|---|------|-------|-----------|--------|------------|-------|
| 1 | Transaction/Account schema designed too narrowly in Phase 1, forcing breaking migrations when Debt/Investments (Phase 3a) need to reuse "account-like" concepts | 1→3a | High | High | Database Architect modeled `Account` with a `type` discriminator and shared balance/interest fields from the start. **Open item restated for 3a kickoff (2026-07-20):** the Database Architect must now explicitly decide whether Debt/Investments are new models referencing an existing typed `Account`, standalone models, or a hybrid — see `roadmap.md`'s Phase 3a section. Not yet resolved. | Database Architect |
| 2 | AI transaction categorization / insights are vulnerable to prompt injection via user-controlled text (merchant names, transaction notes) | 4 | Medium | Medium | Treat all transaction text as untrusted input to the model; structured-output schema validation (Zod) on every AI response before it touches the database; Security Architect review required before release | AI Engineer, Security Architect |
| 3 | CSV import (Phase 1) is a common source of malformed data, encoding issues, and duplicate-transaction bugs | 1 | High | Medium | Backend Engineer validates and dedupes on import; Bug Hunter specifically stress-tests large/malformed CSVs before Phase 1 sign-off | Backend Engineer, Bug Hunter |
| 4 | Financial data (balances, transactions) is high-sensitivity; any authorization bug exposes one user's finances to another | All | Low | Critical | Every API route/Server Action scoped by authenticated user ID at the query level (never trust client-supplied user/account IDs alone); Security Architect reviews every phase before release, not just at the end | Backend Engineer, Security Architect |
| 5 | PDF report generation (Phase 4) and large transaction tables (Phase 1+) risk becoming a performance/memory bottleneck as data grows | 1, 4 | Medium | Medium | Pagination and server-side filtering from day one in Phase 1 (not retrofitted); Performance Engineer reviews before each phase's release. **See Risk #11 — this concern is now concretely realized by Phase 3b's Analytics suite.** | Performance Engineer |
| 6 | Feature scope creep — roadmap is already large; pressure to pull later-phase features into an earlier phase for a "richer demo" | All | High | High | Roadmap phase boundaries are binding; CTO approval required to move any feature to an earlier phase or across the 3a/3b boundary (see Risk #13) | CTO |
| 7 | Better Auth is a comparatively newer library vs NextAuth; ecosystem/adapter gaps could block Phase 0 | 0 | Low | Medium | Time-boxed Phase 0 spike; fall back to NextAuth if Better Auth blocks progress for more than 1 day. **Resolved** — Better Auth shipped cleanly in Phase 0, no fallback needed. | Backend Engineer |
| 8 | Timezone/date handling bugs in bills, budgets (monthly resets), and recurring schedules | 2 | Medium | Medium | Store all dates in UTC, compute "due today"/"month boundary" logic against the user's stored timezone preference, not server time; explicit Bug Hunter pass on this before Phase 2 sign-off | Backend Engineer, Bug Hunter |
| 9 | Debt/Investments schema shape (standalone models vs. extending the existing `Account` type discriminator, vs. a hybrid) decided hastily could force a breaking migration later once 3b's Net Worth aggregation and Analytics depend on it | 3a | Medium | High | Explicit Database Architect design decision as its own reviewed milestone before any Phase 3a backend work starts (not folded silently into implementation) — document the chosen approach and rationale in `docs/database/` before backend implementation begins | Database Architect |
| 10 | Net Worth History chart (3b) has no historical data to chart if periodic snapshotting only begins when the chart UI itself is built | 3a→3b | High | Medium | Begin a lightweight periodic Net Worth Snapshot capture (backend only, no UI) at the end of Phase 3a — see `roadmap.md`'s Phase 3a milestones — so 3b's chart launches with real historical data already accumulated instead of an empty state | Backend Engineer |
| 11 | Analytics suite (3b) is compute-heavy over full Transaction/Account/Debt/Investment history (yearly spend, heatmaps, growth trends) — the general concern already flagged in Risk #5, now realized in concrete form | 3b | Medium | Medium | Performance Engineer reviews query/aggregation design before backend implementation begins (not only at phase-end release review); Solution Architect evaluates materialized/cached aggregate tables during the 3b architecture pass if live-query performance is a concern at realistic data volumes | Performance Engineer, Solution Architect |
| 12 | Financial Goals (broad, 3b) and Savings Goals (Phase 2) are conceptually adjacent — risk of user-facing duplication/confusion or two competing "goal" UIs if the boundary isn't explicit | 3b | Medium | Medium | Product Owner must define the explicit boundary and relationship between the two models in the 3b spec, before Database Architect schema work begins for `FinancialGoal` | Product Owner |
| 13 | Splitting Phase 3 into 3a/3b creates a temptation to blur the boundary (e.g., pulling an Analytics metric into 3a "since it's related," or a Debt/Investment field into 3b) | 3a, 3b | Medium | Medium | The 3a/3b boundary defined in `roadmap.md` is binding, same as Risk #6; CTO approval required to move any feature across it | CTO |

Update this register at the start of each phase's architecture review — add newly discovered risks, close ones that are no longer applicable, and do not delete rows (mark them resolved instead) so the history stays visible to the whole team.
