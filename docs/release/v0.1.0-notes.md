# v0.1.0 Release Notes — Phase 3b (Net Worth History, Analytics, Financial Goals)

**This is the v1 sign-off gate**, per `roadmap.md` ("v1 ships at the end of
Phase 3b") and `project-charter.md`'s Success Definition (as corrected
2026-07-20). Release Manager independent verification, not a re-derivation
of the dispatching summary's claims.

**Decision: BLOCK.** See "Release Manager Decision" at the bottom. This is a
narrow, well-scoped block — one binding, explicit Definition-of-Done
requirement (Analytics test coverage) is unmet and was not disclosed as a
deferral anywhere. Everything else independently re-verified in this pass
(security, the two Bug Hunter fixes, the performance fix, build/typecheck/
lint/tests-as-run, migration state, cross-phase architectural consistency)
is confirmed sound.

## Finding not previously disclosed (the reason for BLOCK)

**`docs/product/analytics.md`'s Definition of Done is not met, and the gap
was not documented as a deferral anywhere** (not in the Phase 3b commit
message, not in the security review, not in a bug report):

> "Yearly Spending, Category Trends, Budget vs. Actual, Income Growth,
> Income Sources, and Savings Growth are all covered by tests verifying
> correct aggregation across month/year boundaries (financial-math-adjacent
> correctness bar, matching the Debt Tracker/Investments/Savings Goals
> specs' own precedent)."
>
> "Savings Growth's investment-gain-adjustment math is verified against
> fixture data including a $0-holdings user and a user with both gains and
> losses in the same period."

Confirmed by direct inspection: `src/features/analytics/server/` contains
exactly one test file, `subscription-detection.test.ts` (covering only
Subscription Cost Detection). There is no test file — and no extracted pure
function to even write one against — for `spending-trends.ts` (Yearly
Spending, Category Trends), `budget-comparison.ts` (Budget vs. Actual),
`income-analytics.ts` (Income Growth, Income Sources), or `savings-growth.ts`
(Savings Growth). Critically, `period.ts` — the one shared module containing
exactly the pure, DB-free month/year-boundary math
(`enumerateMonthKeys`, `resolveMonthKeyRange`, `resolveReportingPeriodRange`)
that every one of those six metrics depends on, and that the DoD explicitly
calls out as needing coverage — has zero tests, despite being trivially
unit-testable with no database (the exact same shape as `payoff-math.ts`,
`progress-math.ts`, and `net-worth-history.ts`'s `resolveRangeStart`/
`thinRows`, all of which *did* get this treatment in this same phase).

This is not a "some judgment call, arguably fine" gap: it is a named,
binding release criterion in the feature's own spec, explicitly modeled on
a bar three sibling Phase 3b/3a features actually met (Debt Tracker's
`payoff-math.test.ts`, Financial Goals' `progress-math.test.ts`, Net Worth
History's `net-worth-history.test.ts`), making Analytics — the largest of
the three Phase 3b components — the one outlier that did not receive the
same rigor, with nobody in the review chain (Backend Engineer, Performance
Engineer, Security Architect, Bug Hunter) flagging or disclosing it as a
deliberate trade-off.

I read the affected code directly (`period.ts`, `savings-growth.ts`,
`budget-comparison.ts`, `income-analytics.ts`) and found no evidence of an
actual defect — the month/year-boundary logic mirrors already-tested
conventions used elsewhere in this codebase (`dashboard/server/service.ts`'s
`resolveMonthToDateRange`/`formatMonthKey`) closely enough that I have no
specific reason to suspect it's wrong. But "the Release Manager eyeballed it
and it looked fine" is precisely the bar the spec's own DoD wording
("covered by tests... not just eyeballed," language it borrows verbatim from
`net-worth-history.md`) says is not good enough for financial-math-adjacent
code, and I am not willing to substitute my own read-through for the
verification the spec itself requires as a release gate.

## What must happen before this gate re-runs

- Backend Engineer: extract the month/year-boundary aggregation math in
  `spending-trends.ts`, `budget-comparison.ts`, `income-analytics.ts`, and
  `savings-growth.ts` into pure, DB-free functions where feasible (mirroring
  `payoff-math.ts`/`progress-math.ts`/`net-worth-history.ts`'s established
  pattern), and add fixture-based tests for `period.ts`'s `enumerateMonthKeys`/
  `resolveMonthKeyRange`/`resolveReportingPeriodRange` at minimum, plus the
  Savings Growth gain/loss-adjustment scenarios the DoD names explicitly
  ($0-holdings user; a user with both gains and losses in the same period).
- Re-dispatch to this Release Manager once tests are added — no other part
  of this gate needs to be redone (security, the two bug fixes, and the
  performance fix are all independently confirmed sound above and do not
  need re-verification unless they change).

## Release Manager Decision

**BLOCK.**

Everything else this gate is responsible for checking — security review,
the two Bug Hunter fixes, the performance fix, tests-as-written passing,
build/typecheck/lint, migration state applied to the real database, and
cross-phase architectural consistency — is independently re-verified and
sound. v1 is *not* being blocked on a suspected functional defect; nothing
found here indicates the app computes anything incorrectly. It is blocked
because one specific, named, binding Definition-of-Done requirement in
`analytics.md` — test coverage for six metrics' month/year-boundary
aggregation math, at the same bar this very phase's other three features
(Debt Tracker, Financial Goals, Net Worth History) demonstrably met — was
silently not delivered, with no documented, reasoned deferral the way the
project's two other known follow-ups (the N+1 query pattern, the Savings
Rate overdue-state gap) properly received. Per the Charter's binding rule
("every release requires: tests passing... enforced by the Release
Manager") and this dispatch's own explicit instruction not to rubber-stamp
a v1 sign-off, I'm not passing this gate on the strength of my own code
read-through standing in for the verification the spec itself requires.

This is a narrow, closeable gap — not a rearchitecture, not a new bug fix,
not a Product Owner decision. Once fixture tests exist for the math this
DoD names, this phase (and v1) should pass without further findings from my
side.
