# FinanceOS — Naming Standards

## Files & folders
- Folders: `kebab-case` (`transactions/`, `data-table/`, `recurring-income/`)
- React component files: `kebab-case.tsx`, default export named in `PascalCase` matching the concept (`account-form.tsx` exports `AccountForm`)
- Non-component TS files: `kebab-case.ts` (`use-accounts.ts`, `validation.ts`)
- Test files: co-located, `<subject>.test.ts(x)`
- **(Phase 3a) Isomorphic pure-calculation files** (importable from both a Server Component and a Client Component — see Architecture.md's guiding-pattern note) live at the feature root, sibling to `types.ts`, never under `server/`. Name them `<concern>-math.ts` (e.g. `features/debt/payoff-math.ts`). These files must never import `lib/db.ts`, `lib/auth.ts`, or anything else server-only.
- **(Phase 3b) Pure-but-server-only calculation files** (no Prisma, fully unit-testable with fixture data, but never called directly by a Client Component) stay under `server/` like any other server file — they do **not** get the feature-root `-math.ts` treatment, since that placement rule is specifically about client-callability, not purity. Name them `<concern>.ts` describing the concern directly (e.g. `features/analytics/server/subscription-detection.ts`, `features/analytics/server/period.ts`), not `-math.ts` — reserving the `-math.ts` suffix exclusively for the feature-root, client-importable case keeps the naming convention itself a reliable signal of placement, not just a style preference.
- **(Phase 4a) AI structured-output Zod schema files: `<concern>-schema.ts`, at the feature's own `server/` directory.** e.g. `features/transactions/server/categorization-schema.ts`, `features/budgeting/server/advisor-schema.ts`, `features/dashboard/server/monthly-summary-schema.ts`, `features/analytics/server/insights-schema.ts`, `features/financial-health-score/server/health-score-narrative-schema.ts`. This suffix is reserved **exclusively** for the Zod schema describing the shape an AI call must return (validated via `lib/ai/generate-structured-output.ts`, per `ai-features-design.md` §3) — it is deliberately distinct from that same feature's existing `validation.ts` (which holds ordinary Server-Action **input** validation schemas, e.g. `CreateAccountSchema`). Phase 4a is the first phase where a single feature's `server/` directory contains two categorically different kinds of Zod schema at once, so this distinction is worth codifying now: a file's suffix alone should tell a reader "this validates what the AI returned" (`-schema.ts`) vs. "this validates what the user submitted" (`validation.ts`), without needing to open the file.
- **(Phase 4a) AI-generation orchestration files: plain `<concern>.ts`, at the feature's own `server/` directory** — no special suffix (e.g. `categorization.ts`, `advisor.ts`, `monthly-summary.ts`, `insights.ts`, `health-score-narrative.ts`). These are ordinary server files that happen to call `lib/ai/generate-structured-output.ts`; they follow the Phase 3b rule above (pure-purpose-named `.ts` files stay under `server/`, never `-math.ts`), extended for the same reason: no AI-generation file is ever a client-instant-recompute target, so none earns the feature-root treatment.

## Code
- Components: `PascalCase` (`AccountForm`, `TransactionTable`, `StrategyComparison`)
- Hooks: `useCamelCase`, always prefixed `use` (`useAccounts`, `useDebts`, `useHoldings`, `useIncomeStreams`, `useNetWorthHistory` — Phase 3b). **(Phase 4a)** no new hooks are introduced — see Architecture.md's Phase 4a Server/client boundary notes for why none of the five AI features needs one.
- Server Actions: verb-first camelCase (`createAccount`, `deleteTransaction`, `createDebt`, `logDividend`, `linkOccurrenceToTransaction`, `dismissSubscriptionCandidate`, `createFinancialGoal`, `archiveFinancialGoal` — Phase 3b; **Phase 4a adds:** `requestCategorySuggestion`, `acceptCategorySuggestion`, `rejectCategorySuggestion`, `refreshBudgetAdvisor`, `refreshSpendingInsights`, `regenerateMonthlySummary`)
- Route Handlers: standard Next.js `GET`/`POST`/`PATCH`/`DELETE` exports, no custom naming. **(Phase 3a exception, documented, not a new rule)**: `app/api/cron/net-worth-snapshot/route.ts` exports `POST` and is authenticated by a shared secret rather than a browser session. **(Phase 3b)** `app/api/dashboard/net-worth-history/route.ts` exports `GET` and follows the ordinary authenticated-session rule (no new exception) — it is simply the first Dashboard Route Handler ever needed, per Architecture.md's Server/client boundary notes. **(Phase 4a)** three more instances of the Phase 3a cron exception — `app/api/cron/categorize-transactions/route.ts`, `app/api/cron/monthly-summary/route.ts`, `app/api/cron/financial-health-score-snapshot/route.ts` — all export `POST`, all shared-secret authenticated, no session. No new session-authenticated Route Handler is introduced this phase.
- Zod schemas: `PascalCase` + `Schema` suffix (`CreateAccountSchema`, `CreateDebtSchema`, `LogDividendSchema`, `CreateFinancialGoalSchema`, `DismissSubscriptionCandidateSchema`, `ReportingPeriodSchema` — Phase 3b); inferred types drop the suffix (`type CreateDebtInput = z.infer<typeof CreateDebtSchema>`). **(Phase 4a)** this same `PascalCase` + `Schema` convention applies equally inside every new `-schema.ts` file above (e.g. `CategorySuggestionSchema`, `BudgetAdvisorRecommendationsSchema`) — the `-schema.ts` **file** suffix (above) and the `Schema` **type** suffix are two independent conventions that compose, not a contradiction: the file tells you *what kind* of schema lives there, the type name still follows the same casing rule every other Zod schema in this codebase already does. The one Feature-1-specific exception: `categorization-schema.ts`'s category-ID enum is **built dynamically per request** (`buildCategorySuggestionSchema(candidateCategoryIds)`, per `ai-features-design.md` §4.2) rather than a single static exported constant — name the builder function itself `build<Concept>Schema`, not `<Concept>Schema`, so its dynamic-per-call nature is visible from the name alone.
- Prisma models: `PascalCase` singular (`Account`, `Transaction`, `Category`, `Debt`, `Holding`, `IncomeStream`, `FinancialGoal`, `FinancialGoalAccount`, `DismissedSubscriptionMerchant`, and — pending the Database Architect's Phase 4a schema pass — the Transaction Auto-Categorization suggestion/audit-trail model, the Budget Advisor/Spending Insights refresh-cache models, and `FinancialHealthScoreSnapshot`) — owned by Database Architect, referenced here for consistency
- Enums (TS and Prisma): `PascalCase` type, `SCREAMING_CASE` members (`AccountType.CREDIT_CARD`). Phase 3a introduced `DebtType`, `AssetType`, `Sector`, `IncomeType`, `IncomeSchedule`, `IncomeOccurrenceStatus` (see prior revisions of this document for their full member lists — unchanged by Phase 3b). Phase 3b introduced `ReportingPeriod`, `SubscriptionStatus`, `FinancialGoalType`, `MeasurementBasis` (see below). **Phase 4a introduces one new internal-only union, no new persisted enum:**
  - `AiFailureReason` (TS-only, internal to `lib/ai/`, never surfaced past `lib/ai/generate-structured-output.ts`'s boundary — per `ai-features-design.md` §5, callers only ever see `AiFeatureResult`'s `"unavailable"` status, never the specific reason): `NETWORK_ERROR | TIMEOUT | VALIDATION_FAILED | GROUNDING_FAILED`, `SCREAMING_CASE` members per the existing enum convention, used only for internal logging/observability, never persisted and never part of any exported API contract.
  - No new **persisted** enum is introduced this phase — the suggestion/audit-trail table's lifecycle state and the Financial Health Score's banded label are candidates the Database Architect may formalize as enums during their schema pass; if so, they follow this same `PascalCase` type / `SCREAMING_CASE` member convention, consistent with every enum in this schema to date.
- **URL searchParam values use `kebab-case`**, matching the existing convention already established by Budgeting's `?month=YYYY-MM` and Bills' `?view=list|calendar`: Analytics' reporting-period control is `?period=this-year|last-12-months|year-to-date|all-time`, and the Net Worth History chart's range selector is `?range=30d|90d|1y|all` (the client-side hook's query param, not a page searchParam, since the chart refetches via `use-net-worth-history.ts` rather than a full navigation — see Architecture.md's Phase 3b data-flow example). **(Phase 4a)** no new searchParam is introduced — every one of the five features' controls (month, reporting period) reuses an existing convention already established above; Spending Insights on the Analytics page reuses `?period=` verbatim, per `ai-features.md` AC5.

## AI-specific conventions (Phase 4a)

These conventions are new this phase and apply uniformly across all five AI features — see `docs/architecture/ai-features-design.md` for the full behavioral reasoning behind each; this section only standardizes the *names*.

- **`AiFeatureResult<T>`** — defined once, in `lib/ai/types.ts`:
  ```ts
  type AiFeatureResult<T> =
    | { status: "ok"; data: T }
    | { status: "unavailable" }
  ```
  This is the **mandatory** return type of every feature-owned AI-generation function (`categorization.ts`, `advisor.ts`, `monthly-summary.ts`, `insights.ts`, `health-score-narrative.ts`) — never a thrown exception, per `ai-features-design.md` §5's "Result type, not a thrown error" reasoning. No feature may define its own parallel "maybe it failed" shape (e.g. no `{ success, data } | { success, error }` reuse of `ApiResult<T>` for this purpose, and no feature-specific `CategorizationResult`/`AdvisorResult` union) — `AiFeatureResult<T>` is the **one** shared shape, generic over the feature's own payload type, exactly the way `ApiResult<T>` is already the one shared shape for ordinary request outcomes. Reusing `ApiResult<T>` here instead would conflate two different questions ("did the request succeed" vs. "did the AI produce usable output") into one boolean, which is exactly the ambiguity `ai-features-design.md` §5 introduced this type to avoid.
- **Composing `AiFeatureResult<T>` with `ApiResult<T>`** — every on-demand (Server Action) AI feature returns `ApiResult<AiFeatureResult<T>>`, **never** `AiFeatureResult<ApiResult<T>>` (the outer layer is always the transport-level result; the AI-specific outcome always nests inside `data`). This ordering is fixed and non-negotiable across all five features so a caller never has to remember which feature nested them which way. See api-contracts.md's Phase 4a section, "The `ApiResult<AiFeatureResult<T>>` composition," for the full read-once explanation and why a degraded AI outcome is `{ success: true, data: { status: "unavailable" } }`, never `{ success: false, ... }`.
- **Server-Component-direct-call reads that surface AI-generated content return `AiFeatureResult<T>` directly**, with no `ApiResult` wrapper at all — consistent with every other Server-Component-direct-call row in this codebase never being `ApiResult`-wrapped (e.g. Accounts' plain `Account[]`). Only Server Actions and cron Route Handlers ever produce an outer `ApiResult`/plain-JSON layer.
- **The Financial Health Score's deterministic half is the one AI-adjacent surface that is never `AiFeatureResult`-wrapped.** `service.getFinancialHealthScore(userId)` returns a plain `FinancialHealthScoreBreakdown` — not `AiFeatureResult<FinancialHealthScoreBreakdown>` — because the score has zero AI dependency by design (`ai-features.md` Feature 5's own resolved scope). Only `service.getLatestNarrative(userId)` (the optional explanatory text) is `AiFeatureResult`-wrapped. Naming this distinction matters: a reviewer should be able to tell, from the return type alone, that the score can never be affected by an AI outage — wrapping it "for consistency" would misrepresent that guarantee.
- **Model-tier constants**: `fastModel` / `reasoningModel`, exported from `lib/ai/client.ts`, lower-camelCase — consistent with this codebase's existing lower-camelCase singleton-export convention (`utapi` in `lib/uploadthing.ts`, `db` in `lib/db.ts`). Never referred to by a raw model-ID string anywhere outside `client.ts`.
- **Rate-limit helper naming**: `canRefreshNow(lastGeneratedAt, minIntervalMs): boolean`, in `lib/ai/rate-limit.ts` — a plain, verb-first predicate function name, consistent with this codebase's existing predicate-naming convention (`isCompleted`, `isOverBudget`, `isNegativeAmortization`). Every one of the five features' on-demand refresh/reconsider paths calls this exact function; no feature defines its own differently-named rate-limit check.
- **Cache-row field naming for the Budget Advisor/Spending Insights refresh caches** (schema TBD, Database Architect): whatever the final column name, the field recording when content was last generated must be named `generatedAt` (not `updatedAt`, `lastRefreshedAt`, or any other synonym) across both features' cache rows, since `rate-limit.ts`'s `canRefreshNow` takes a `lastGeneratedAt` parameter by that same name — keeping the concept's name consistent from the database row through to the function signature that reads it avoids a silent renaming seam between schema and code.

## API response shape (every Route Handler and Server Action)
```ts
type ApiResult<T> =
  | { success: true; data: T }
  | { success: false; error: string };
```
Defined once in `lib/api-response.ts`. No endpoint invents its own shape. **(Phase 3a)** `app/api/cron/net-worth-snapshot/route.ts` is the sole documented exception. **(Phase 3b)** `app/api/dashboard/net-worth-history/route.ts` follows the ordinary `ApiResult<T>` rule (it is a normal, session-authenticated, client-facing endpoint — no exception here, only `net-worth-snapshot`'s system-to-system cron route is exempt). **(Phase 4a)** three more system-to-system cron routes join `net-worth-snapshot`'s exception (`categorize-transactions`, `monthly-summary`, `financial-health-score-snapshot`) — all plain JSON, no session, no `ApiResult<T>`. Every other Phase 4a Server Action follows `ApiResult<T>` exactly as before, with the one addition of the `AiFeatureResult<T>` nesting documented above (a composition on top of `ApiResult<T>`, not an exception to it).

## CSS / Tailwind
- No inline arbitrary colors — use the Tailwind theme tokens configured for shadcn/ui (`bg-primary`, `text-muted-foreground`, etc.) so theming/dark mode stays centralized.
- Category/account "color" fields (user-chosen) are stored as hex strings in the database and applied via inline `style`, not Tailwind classes — the only sanctioned exception, since these are user data, not design tokens.
- **(Phase 3a)** Gain/loss and negative-amortization warnings use the same red/green semantic-color convention already established for over-budget indicators (Phase 2).
- **(Phase 3b)** Net Worth History's Assets/Debt breakdown series, Savings Growth's chart, and Subscription Cost Detection's Active/Possibly Cancelled badges all reuse this same existing red/green/neutral semantic-color token set — no new ad hoc colors are introduced for any Phase 3b feature specifically, same discipline already applied to Phase 3a's gain/loss and negative-amortization indicators.
- **(Phase 4a)** The "AI-generated" visual label required by `ai-features.md`'s Cross-Cutting Requirement #3 (marking a suggestion/recommendation/summary/insight/narrative as AI-authored, distinct from a verified computed number) is a **UI Component Engineer** concern — one shared, reusable badge/icon component, not five independently-styled labels — but its exact visual spec (icon choice, color token, placement) is out of this Architect's and this document's scope to dictate; flagged here only so the naming/placement decision (one shared component, reused by all five surfaces) is recorded, not the visual design itself.
